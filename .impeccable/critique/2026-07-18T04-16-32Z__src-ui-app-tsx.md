## Anti-Patterns Verdict

**LLM Assessment**: This passes the AI slop test convincingly. The dark precision-instrument palette with film-stock amber accent is a deliberate, domain-appropriate choice — not the "dark mode because tools look cool" reflex. No cream/sand backgrounds, no gradient text, no glassmorphism, no hero metrics, no identical card grids, no eyebrow text, no numbered section markers, no side-stripe borders. The amber accent is genuinely distinctive: mid-tier NLEs overwhelmingly use blue or cyan accents (Premiere = purple/blue, DaVinci = blue-gray, Final Cut = gray, CapCut = pink/magenta). Amber says "craft, film stock, light meters" — it carries meaning.

The warmth in the surface stops (the undertone across ink→plate→panel→elevated) prevents the sterile digital void that plagues most dark-mode tools. This is a tool that feels inhabited, not generated.

**Deterministic Scan**: 46 findings, all advisory:

- **43 color-drift findings**: Undocumented colors in `App.tsx` (HSLA, hex, named colors like `#FFD700`, `#0A0F1E`). These are likely inline styles on SolidJS components that have drifted from the token system. Severity: advisory — these are real colors rendering in the UI that exist outside the design system's governance.
- **3 radius-drift findings**: `border-radius: 4px` outside the DESIGN.md rounded scale in `global.css`. Severity: advisory.

These are not bugs; they're signals that the codebase has accumulated visual decisions that bypassed the token layer. That's normal for a 48-phase project, but the advisory count is high enough to note.

**Visual Overlays**: Browser injection was not available (assessment B agent produced no output). A live browser inspection would be valuable for a follow-up pass — the dev server at `http://localhost:5173` is running and accessible.

## Overall Impression

This is a remarkably disciplined workspace for a project of this scale. Three columns, one purpose each: library → canvas → properties. The layout follows 25+ years of NLE convention without being derivative — the amber accent, warm undertone, and flat hairline panels give it its own voice. The biggest opportunity is not restructuring but **reinforcing**: the token system is strong but shows drift, the rail navigation is at its cognitive ceiling, and the collapsed/empty states leave users guessing.

## What's Working

1. **The three-column grid is the right call.** Library (236px) | Canvas (1fr) | Inspector (304px) maps directly to the editor's mental model: source → work → properties. The responsive collapse at 900px to a single-column flex layout is thoughtful — below that width, a 3-column NLE grid stops being useful anyway.

2. **The amber accent carries genuine personality.** `#d4a853` on `#0a090f` hits 9.0:1 contrast, is warm without being garish, and stands apart from the blue/cyan/magenta monoculture of competing NLEs. The `amber-soft` token (`rgb(212 168 83 / 10%)`) as an active-tab background tint is a nice touch — it signals selection without shouting.

3. **The tab system is structurally sound.** Primary tabs (Inspector, Text, Audio, Capture) group related tools by user job, not by implementation layer. Secondary segmented controls (Live Chain | Voice FX, Record | Program | Replay | Go Live) add a second dimension of organization without adding a third navigation tier. This is information architecture done right.

## Priority Issues

### [P1] Panel title text fails WCAG AA contrast at 9.5px

- **What**: `.panel-title` uses `var(--ink-700)` (#5a5c70) on `var(--panel)` (#16151c) at 9.5px font size — contrast ratio is **2.76:1** (needs 4.5:1 for small text).
- **Why it matters**: Panel titles are the primary signposts in dense professional UIs. When they're unreadable at a glance, users scan past them and rely on position memory alone. This is a real accessibility failure, not a nit.
- **Fix**: Bump `.panel-title` color to `var(--ink-500)` (#8e90a3, 5.76:1 contrast) or `var(--ink-400)` (#a8aab8, 7.86:1). The current `ink-700` (#5a5c70) should be reserved for purely decorative elements, never for functional labels.
- **Suggested command**: `/impeccable audit src/global.css`

### [P1] Collapsed side rail has no recognizable affordance

- **What**: When the right rail is collapsed, the only visible element is a 28px-wide button containing the single character `‹` (U+2039, Single Left-Pointing Angle Quotation Mark). The button has an `aria-label` and `title` but no visible text label.
- **Why it matters**: A first-time user or someone returning after a break sees `‹` and has zero information about what it expands. Jordan (first-timer) would never click it. Even Alex (power user) relies on spatial memory. This is a recognition-over-recall failure.
- **Fix**: Replace `‹` with a vertical label that reads the current active tab name (e.g. "Inspector") rotated or abbreviated, or use a more recognizable icon. At minimum, add a visible tooltip on hover.
- **Suggested command**: `/impeccable clarify side-rail`

### [P2] 46 instances of color drift erode the token system

- **What**: The detector found 43 undocumented colors in `App.tsx` and 3 undocumented radii in `global.css`. These are hardcoded values that bypass the design token layer.
- **Why it matters**: Each hardcoded color is a future maintenance hazard. When the palette shifts (e.g., a dark-mode refinement), hardcoded values don't update. Over time, the UI develops visual inconsistency — slightly different ambers, slightly different grays, each "close enough" to pass review but collectively muddying the precision.
- **Fix**: Audit the top offenders (most frequently used undocumented colors), add them to the DESIGN.md palette or replace them with existing tokens. The `#FFD700` (gold) is a particularly visible example — is it a second accent, or should it be `var(--amber)`?
- **Suggested command**: `/impeccable document`

### [P2] Right rail navigational depth hits the working memory ceiling

- **What**: The user must navigate 4 primary tabs, each with 2–4 secondary tabs. The Capture primary tab alone presents 4 sub-tabs (Record, Program, Replay, Go Live). That's 4 × 4 = 16 possible destinations behind a two-tier tab system.
- **Why it matters**: With 4 primary tabs and up to 4 sub-tabs, the user must hold two levels of navigation state in working memory. After switching from Audio → Voice FX to Capture → Replay, the user has to remember where they were. This is a moderate cognitive load violation (decision points at exactly the 4-item ceiling).
- **Fix**: Consider collapsing Replay into Record (they're both capture-time features) to bring Capture to 3 sub-tabs. Or consider whether "Go Live" (WHIP publish) belongs under Capture or should be a standalone toolbar action — streaming is a distinct user intent from recording.
- **Suggested command**: `/impeccable distill side-rail`

### [P3] No visible empty-state guidance in the workspace

- **What**: Before media is imported, the workspace likely shows a dark canvas with no guidance. The preview panel has "calibration-grid preview empty state with cyan corner brackets" (per the post-merge chrome cleanup spec), but other panels (Inspector, Audio, Text) in their empty state are unknown.
- **Why it matters**: Jordan (first-timer) opens the app and sees panels full of nothing. Without an empty-state cue ("Import media to begin" or "Select a clip to edit properties"), the interface feels broken rather than waiting.
- **Fix**: Add purpose-stating empty states to the Inspector ("Select a clip to edit its properties"), Text tab ("Import captions or use Auto Captions"), and Audio tab ("Audio tools activate during playback").
- **Suggested command**: `/impeccable onboard`

## Persona Red Flags

### Alex (Power User)

- **No visible keyboard shortcut reference.** The toolbar has undo/redo/split/delete buttons but no shortcut hints. Alex expects to see `⌘Z` on hover or in a cheatsheet. The project has a keyboard map (Phase 10), but it's not surfaced in the UI.
- **No batch operations in the media bin.** Alex wants to select 5 clips, right-click, "Add to timeline" — or apply a common effect to multiple clips. The inspector is single-clip only.
- **Collapsed rail is unreachable by keyboard.** The `‹` button may not be in a natural tab order. Alex expects `Cmd+Shift+[` or similar to toggle the rail without touching the mouse.

### Sam (Accessibility-Dependent)

- **Panel title text fails WCAG AA.** The `.panel-title` at 2.76:1 contrast is below the 4.5:1 requirement for text this small. Screen-reader users won't notice, but low-vision users will.
- **Tab text at 9.5px may be too small at 200% zoom.** At 200% zoom on a 1440px display (effectively 720px viewport), the tabs need to remain legible. The workspace already collapses to single-column at 900px, but the tab sizing in that collapsed state isn't visible in the CSS I reviewed.
- **Collapse/expand buttons use Unicode characters (`‹`, `›`) as the sole visible label.** Screen readers will pick up the `aria-label`, but the visual presentation fails users who can see but not read tiny punctuation.

### Riley (Deliberate Stress Tester)

- **What happens when all 4 Capture sub-tabs are open simultaneously?** The `lazyMount` + `unmountOnExit` on Tabs.Root helps, but if a user rapidly switches between tabs with heavy content (Inspector with a selected clip, Capture with streaming controls), does memory pressure spike?
- **The color-drift findings suggest inconsistent rendering.** If one component uses `#FFD700` (hardcoded gold) and another uses `var(--amber)`, a palette change would update one but not the other. Riley would notice the inconsistency in a side-by-side comparison.
- **Empty workspace state before any media import.** Refresh the page with no project loaded — does every panel degrade gracefully, or do some throw console errors?

## Minor Observations

- The `.dock-left` overflow is set to `hidden` with no scroll affordance. If the Beat Panel content is taller than the viewport, it's simply clipped.
- The side-rail collapse/expand buttons use `var(--font-mono)` (JetBrains Mono) for the `‹`/`›` glyphs. A mono font is a strange choice for a directional arrow — the glyph is designed for quotation, not navigation.
- The workspace gap is 6px — tight and deliberate for a tool UI, but the padding is also 6px. This creates equal spacing between panels and between panels and the viewport edge, which flattens the visual grouping. Consider 6px gap + 8–10px outer padding for a subtle frame-within-frame effect.
- The toolbar sits outside the workspace grid in the app shell — a separate full-width strip. This is correct architecturally (toolbar = global, workspace = contextual), but visually it can feel disconnected from the three-column layout below.

## Questions to Consider

1. **What if the right rail only showed 3 primary tabs?** Inspector + one contextual tab based on the current selection (Audio when a clip is selected with audio, Text when a caption track is active, Capture as a standalone action). This would reduce from 4 to ≤3 primary tabs and collapse the secondary tab tier into conditional primary tabs.

2. **What would a confident empty workspace look like?** Instead of dark panels waiting to be filled, what if the workspace greeted the user with a single bold action ("Import Media" or "Start Recording") and progressively revealed complexity as media arrived?

3. **Does the Capture tab earn its spot in the primary rail?** Recording and streaming are fundamentally different from editing — they're about acquisition, not arrangement. What if Capture moved to a toolbar button that opened a dedicated overlay or split the preview into a recording view, leaving the right rail to 3 purely editing-focused tabs?
